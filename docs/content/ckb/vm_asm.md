# CKB/CKB-VM 深入了解汇编执行器的实现

在今年早些时候, 我们为 CKB-VM 引入了一个新的汇编执行器, 以提高 CKB-VM 在 RISC-V IM 指令集上的性能. 这个[新的汇编执行器](https://github.com/nervosnetwork/ckb-vm/pull/502)使用 RISC-V 汇编语言手工编写, 以最大化的利用 RISC-V 架构的特性以提升性能. 趁此机会, 我想为大家介绍一下 CKB-VM 里的各个汇编执行器的技术细节, 以及更重要的: 如果你想为 CKB-VM 编写一个新的汇编执行器, 你需要了解哪些知识.

## 为什么还需要汇编执行器

CKB-VM 本质上是一个 RISC-V 虚拟机. 最直接的实现方式当然是解释器: 取一条 RISC-V 指令, 解码, 执行, 更新 PC, 然后继续下一条. 这个模型非常清晰, 也很适合作为正确性的基准. 但问题在于, 对许多链上脚本而言, 单条指令本身做的事情很小, 解释器为了执行这条指令付出的固定开销反而很大.

想象一个最简单的 `addi` 指令. 它的语义只是把一个寄存器加上立即数, 再写回目标寄存器. 真正的计算可能只需要宿主 CPU 的一两条指令, 但普通解释器要先进入 Rust 的 match 分支, 读取寄存器, 处理零寄存器, 维护周期, 再跳回循环顶部. 当这样的动作重复几千万次时, 这些开销就不再是背景噪音, 而是变成性能消耗的主体.

汇编执行器要解决的正是这件事: 把解释器最热的那一圈循环, 改写成宿主架构上最直接的机器动作.

这里容易产生一个误解: CKB-VM 的 ASM 后端并不是 JIT. 它不会把用户的 RISC-V 程序翻译成一段新的宿主机器码再执行. 它仍然是解释器, 只是解释器的调度循环, 寄存器读写, 内存检查这些热点路径都由手写汇编完成. 如果说普通解释器像一位照着菜单做菜的厨师, 那 ASM 后端就是同一个厨师把最常用的动作练成了肌肉记忆.

## Trace: 把分派成本提前付掉

CKB-VM 的 ASM 执行器并不是每次都从原始指令开始解码. 在真正进入汇编循环之前, VM 会先把一段指令整理成 trace. trace 中的每个条目都包含两部分信息:

```text
+----------------------+----------------------+
| 下一段汇编标签地址   | 已经整理好的操作数   |
+----------------------+----------------------+
```

在汇编文件里, 这个模型被压缩成两个连续前进的指针: `INST_PC` 指向下一条指令对应的宿主汇编标签, `INST_ARGS` 指向这条指令的参数. 每执行完一条指令, `NEXT_INST` 宏就把两者各自向前推进 16 字节, 取出下一条指令的标签地址, 然后直接跳过去.

以 RISC-V 版本为例, `NEXT_INST` 的形状大致如下:

```asm
#define NEXT_INST \
	ld     t0, (INST_ARGS); \
	ld     t1, (INST_PC); \
	addi   INST_ARGS, INST_ARGS, 16; \
	addi   INST_PC, INST_PC, 16; \
	srli   RD, t0, 8; \
	andi   RD, RD, 0xff; \
	srai   INST_HI32, t0, 32; \
	jr     t1
```

这段代码很短, 但它是整个 ASM 后端的心脏. `jr t1` 不是跳回一个巨大的 `match`, 而是直接跳到 `.CKB_VM_ASM_LABEL_OP_ADD`, `.CKB_VM_ASM_LABEL_OP_LD_VERSION1`, `.CKB_VM_ASM_LABEL_OP_ECALL` 这样的标签. 原本每条指令都要付出的分派成本, 被提前折叠进 trace 构建阶段. 执行阶段只剩下"拿地址, 跳过去".

这也是为什么新增一个汇编执行器时, 不能只盯着每条指令怎么写. 你首先要理解 CKB-VM 怎样把一条 RISC-V 指令变成 trace 中的 16 字节条目, 以及 `ckb_vm_asm_labels` 怎样把 opcode 映射到汇编标签. 指令本身只是树叶, trace 才是树干.

当前源码的 trace 模块里, 除了默认使用的 fixed trace, 还提供了 dynamic trace 这一支实现. fixed trace 是一张固定大小的表, 通过 `(PC >> 2) & mask` 找到槽位; dynamic trace 则用于更长的顺序代码块. 有趣的是, 这两种 trace 在汇编层看起来几乎一样: 前面都是 `address`, `length`, `cycles`, 后面跟着同样的 16 字节 thread 序列. 当对应的 trace decoder 发现 fixed trace 后面还有一段连续代码时, 它会把下一条 thread 改成 `OP_CUSTOM_ASM_TRACE_JUMP`, 让汇编执行器直接跳到 dynamic trace. Rust 侧的数据结构变复杂了, 但汇编侧仍然只认同一种形状, 这正是这个设计漂亮的地方.

## 寄存器分配: 汇编执行器的骨架

写 Rust 时, 我们很少直接关心变量住在哪个寄存器里. 写汇编执行器时, 这件事反过来成了第一道门槛. CKB-VM 的三份执行器一开头都在做同一件事: 给机器状态, trace 指针, 指令操作数和临时变量安排宿主寄存器.

x64 版本中, `MACHINE` 放在 `%rsi`, `TRACE` 放在 `%rbx`, `INST_PC` 和 `INST_ARGS` 分别放在 `%r8` 和 `%r9`. `RD`, `RS1`, `RS2` 等 RISC-V 操作数也各自绑定到固定的 x86-64 寄存器上. aarch64 版本则使用 `x19` 到 `x29` 这一组寄存器保存长期状态. 新增的 riscv64 版本里, `MACHINE` 使用 `a0`, `INVOKE_DATA` 使用 `a1`, 而 `RD`, `RS1`, `RS2`, `RS3`, `IMMEDIATE` 被安排到 `s0` 到 `s5`.

这种分配看似只是命名, 实际上决定了后面几千行代码的难度. 例如 x64 的除法指令天然依赖 `%rax` 和 `%rdx`, 移位指令又要求 shift count 放在 `%cl`. 因此 x64 文件开头会特别注明: 哪些虚拟寄存器不能占用 `%rcx`, 哪些不能占用 `%rax`, 哪些不能占用 `%rdx`. 这不是洁癖, 而是为了避免后面每实现一条 `div` 或 `sll` 都要在寄存器之间来回搬运.

RISC-V 宿主版本在这点上有一种很有趣的对称性: 我们正在用 RISC-V 汇编解释 RISC-V 指令. 但这并不意味着可以直接把 guest 寄存器映射到 host 寄存器. CKB-VM 的 guest 有 32 个通用寄存器, 宿主可自由使用的寄存器远远不够, 更不用说还要保存机器状态, 内存指针和 trace 指针. 所以 RISC-V ASM 后端仍然只把当前指令需要的几个操作数放进宿主寄存器, 执行完再写回 `AsmCoreMachine` 的寄存器数组.

换句话说, 它不是"让 RISC-V 跑 RISC-V", 而是"用 RISC-V 写一个非常薄的解释器".

## 零寄存器的小麻烦

RISC-V 的 `x0` 永远等于 0. 这条规则很简单, 但在执行器实现里非常容易变成小陷阱.

在早期版本中, CKB-VM 的 ASM 后端常常在写回 `rd` 之后, 再把寄存器数组里的第 0 项清零. 所以我们会看到类似这样的宏:

```asm
#define WRITE_RD(reg1, tmp1) \
	LOAD_REGISTER_ADDRESS(RD, tmp1); \
	sd     reg1, CKB_VM_ASM_ASM_CORE_MACHINE_OFFSET_REGISTERS(tmp1); \
	sd     zero, CKB_VM_ASM_ASM_CORE_MACHINE_OFFSET_REGISTERS(MACHINE)
```

也就是说, 即使某条指令误把结果写到了 `x0`, 下一步也会把它擦掉. 后来一些 MOP 伪指令会在一条 guest 语义中写多个寄存器, 这时如果每写一次都立刻清零 `x0`, 写回顺序本身就可能影响结果. 因此源码里又出现了 `WRITE_RD_V2` 和 `NEXT_INST_V2` 这一组宏: 前者只负责写目标寄存器, 后者在整条伪指令结束时统一清零 `x0`. 从这个细节可以看出, 汇编执行器不只是在实现 RISC-V 指令本身, 它还要背着 CKB-VM 各个历史版本的语义包袱前进.

这类包袱在 `LOAD_PC`, `JALR_VERSION0/1`, `LB_VERSION0/1` 等标签中都能看到. 例如内存读取的边界判断, version 0 和 version 1 对"刚好落在末尾"的处理并不完全相同. 普通 Rust 代码里, 我们可以把这种差异藏在一个函数里; 汇编执行器里, 它会变成两个相邻的标签, 或者一个带版本参数的宏.

## 内存检查: 最热也最不能省的路径

如果只看算术指令, 汇编执行器并不难理解. `add` 就是 load 两个寄存器, 加起来, 写回. 真正让它变复杂的是内存.

CKB-VM 的内存模型既要保证越界访问会被拒绝, 又要维护 W^X 权限, 还要支持按需初始化的 FastMemory. 每次 `lb`, `lw`, `ld`, `sb`, `sw`, `sd` 都可能触发这一整套逻辑. 如果这些检查写得太慢, ASM 后端的大部分收益都会被吃掉; 如果写得太激进, 链上执行的确定性和安全边界就会出问题.

三份汇编执行器都使用了类似的策略: 先记住上一次读过的 memory frame, 或上一次写过的 page. 如果当前访问仍然落在同一个范围内, 就直接走快速路径; 如果跨过边界, 才进入完整检查.

读路径大致分为三步:

- 检查地址加长度是否越界.
- 查看对应 memory frame 是否已经初始化.
- 如果没有初始化, 调用 Rust 侧的 `inited_memory` 完成实际填充.

写路径则多了一层权限判断: 对应页面必须是 writable, 同时要把 dirty bit 置上, 以便快照和后续内存管理知道这一页被修改过.

这里最值得注意的是 `PREPCALL` 和 `POSTCALL`. 汇编执行器大部分时间都不想回到 Rust 世界, 但内存初始化这类工作仍然由 Rust 侧负责. 一旦需要调用 `inited_memory`, 汇编代码就必须保存当前寄存器状态, 按目标平台 ABI 对齐栈, 调用函数, 再恢复现场. PR 502 中专门有一个提交叫做 "Sp should always be 16-byte aligned", 说的就是这个细节. 它听起来像一条普通的 ABI 规则, 但在手写执行器里, 这类规则一旦漏掉, bug 往往会以非常诡异的形式出现.

## 三个宿主架构, 同一套语义

现在 CKB-VM 中有 x64, aarch64, riscv64 三份汇编执行器. 它们长得很像, 但气质完全不同.

x64 版本最早出现, 也最像一位经验丰富但有点挑剔的老工匠. 它能用 `imul`, `idiv`, `bsr`, `bsf` 这类指令把很多 RISC-V 语义压得很紧, 但代价是寄存器约束复杂. 尤其是除法, 商和余数天然落在 `%rax`/`%rdx`, 这让周围的保存和恢复都必须非常小心.

aarch64 版本寄存器更宽裕, 条件选择和位提取指令也好用. 例如 `ubfx` 可以很自然地从 trace 参数中切出 `rd`, `rs1`, `rs2`; `csel` 可以减少一些分支. 但它也有自己的 ABI 细节, 比如在 macOS 下符号名前缀和 section 写法都要特殊处理.

riscv64 版本则有一种"回到母语"的感觉. `add`, `sub`, `mul`, `sraiw` 这些 guest 指令, 在宿主上往往可以找到非常接近的对应写法. 但它也不是一路绿灯. RISC-V 立即数加载, 地址计算, 调用约定, 以及没有某些高阶位操作指令时的软实现, 都会让代码变得比想象中更长. 例如 B 扩展中的 `clmul`, `clmulh`, `clmulr` 在 riscv64 ASM 后端里仍然要通过循环和位运算模拟, 因为执行器不能假设宿主 CPU 一定支持同样的扩展.

这就是多后端执行器最重要的原则: **宿主架构可以不同, guest 语义必须完全一致**. 性能优化只能发生在这个边界之内.

## PR 502: 从一份新文件到一个新后端

PR 502 的标题很朴素: 让 ckb-vm 能在 rv64im 环境中使用 assembly mode. 如果只看变更统计, 最显眼的当然是新增了 `execute_riscv64.S`: 1900 多行手写汇编, 覆盖基础 IM 指令, A 扩展, B 扩展, MOP 伪指令, 内存检查和退出路径.

但这个 PR 真正值得学习的地方, 是它没有把"新增执行器"理解成"提交一个 `.S` 文件". 它同时做了几件配套工作.

第一件事是让构建系统认识 riscv64. `build.rs` 里原本只判断 x86_64 和 aarch64, PR 中新增了 riscv64 分支. 发展到当前源码, 这段逻辑已经收束得更清楚: 先判断目标架构是否属于 `aarch64`, `riscv64`, `x86_64`, 再在 `asm` 或 `detect-asm` feature 开启时, 按 `target_arch` 选择 `execute_aarch64.S`, `execute_riscv64.S` 或 `execute_x64.S`. 这说明新增后端的第一步不是写指令, 而是让它在正确的平台上被编译, 在错误的平台上保持沉默.

第二件事是整理标签表. CKB-VM 的 trace 里保存的是跳转目标, 而这些目标来自 `ckb_vm_asm_labels`. PR 中把标签表放进只读 section, 并把 label offset 的基准从标签表自身调整为执行入口. 当时这个入口还叫 `.CKB_VM_X64_EXECUTE`, 后来随着命名统一改成了现在的 `.CKB_VM_ASM_EXECUTE`. 这个改动看起来绕了一点, 但它解决的是多架构共享 label table 表达方式的问题: 表里不直接塞绝对地址, 而是塞相对于一个稳定入口的偏移. Rust 侧再通过执行入口地址把它还原成可跳转的目标.

第三件事是测试策略. PR 描述里特别提到, 过去 CI 会在流程中构建 test cases, 但在 riscv64 上构建这些用例很麻烦, 所以改成把 ckb-vm-contrib 中的测试用例打包成 artifact 放进仓库. 新增的 `test_artifact.rs` 会把 `tests/artifact/arch`, `tests/artifact/cryptography`, `tests/artifact/spec` 中的程序分别拿给解释器和 ASM 后端执行, 确认它们都能成功返回 0. 这之外, 仓库里还有 A 扩展, B 扩展, MOP, 版本差异等更细的专项测试, 用来覆盖那些 artifact 不一定能踩到的边界.

这一步很重要. 手写汇编执行器最怕的不是完全跑不起来, 而是只在某个边界条件上偏离解释器. 一个 `lb` 的末尾边界, 一个 `sc.w` 的 reservation 地址, 一个 `adc` 的写回顺序, 都可能让绝大多数程序通过, 但在某个密码学实现里埋下不一致. artifact 测试的价值, 就是把大量真实指令组合压到同一条等价性跑道上.

第四件事是 review 过程中不断收紧的小修正. PR 后续提交里有几个名字很能说明问题: `Adjust ADC and SBB's writing order`, `Use lwu to load trace length`, `Fixing a boundary issue in FastMemory`. 这些都不是宏大的架构设计, 却是汇编执行器能否真正落地的关键. 因为到了这一层, bug 往往不是"少实现了一条指令", 而是"这条指令在某个寄存器别名场景下早写了一步".

## 如果要写一个新的汇编执行器

假设明天我们要为另一个架构新增 ASM 后端, 我不会从 `ADD` 开始写. `ADD` 太容易让人产生错觉: 好像只要把指令表翻译一遍就完成了. 真正可靠的起点应该是执行器的空壳.

首先要让入口函数能被 Rust 调到, 能拿到 `MACHINE` 和 `INVOKE_DATA`, 能正确保存宿主 ABI 要求的 callee-saved 寄存器, 能在暂停, ecall, ebreak, 越界, 权限错误等情况下回到 Rust. 这一步完成后, 哪怕所有指令都走 slowpath, 后端也已经有了生命体征.

然后要打通 trace. 你需要让 `ckb_vm_asm_labels` 能表达这个架构的标签地址, 让 `NEXT_INST` 可以稳定取出下一条指令的跳转目标和参数, 让 `CUSTOM_TRACE_END` 可以正确加载 trace, 累加 cycles, 检查 max cycles. 只有这条循环稳定后, 后面几百个 opcode 才有地方安放.

接着才轮到指令本身. 我会先实现最小的 IM 子集: `addi`, `add`, `sub`, `jal`, `jalr`, `lb/lw/ld`, `sb/sw/sd`, `ecall`. 这组指令足以跑起很多小程序, 也足以暴露寄存器写回, PC 更新, 内存检查和退出路径的问题. 等这条路径稳定后, 再扩展到 A, B, MOP 等更复杂的部分.

最后是测试. 汇编执行器的测试目标不应该是"这个后端能跑几个程序", 而应该是"它和参考解释器在同一批程序上给出同样结果". PR 502 的 artifact 测试正是这个思路. 真实程序, 官方 spec case, 密码学程序, 再加上一些专门构造的边界样例, 才能把执行器从"看起来正确"推向"足够可信".

## 小结

CKB-VM 的汇编执行器是一种很克制的优化. 它没有引入 JIT, 没有改变链上程序的语义, 也没有把安全边界交给宿主机器码生成器. 它做的是更朴素也更辛苦的事: 把解释器最热的路径逐条翻译成手写汇编, 同时保留 CKB-VM 原有的内存模型, 版本语义, 周期计量和异常行为.

从 x64 到 aarch64, 再到 PR 502 中的 riscv64, 这条路线展示了一种很有 CKB-VM 风格的工程取舍: 先有清晰的参考实现, 再在热点路径上做可验证的局部替换. 每新增一个后端, 都不是另起炉灶, 而是在同一套语义之下, 用目标架构自己的语言把那圈循环重新讲一遍.

这也是我最喜欢这个设计的地方. 汇编执行器看起来离业务很远, 满屏都是寄存器, 标签和 ABI 细节. 但它真正服务的目标很简单: 让链上程序在不牺牲确定性和安全性的前提下跑得更快. 对一个区块链虚拟机来说, 这已经是足够朴素, 也足够困难的目标了.
